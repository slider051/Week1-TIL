# I AM — 나만의 공간 업그레이드 (진행 가이드)

> 사용 도구: Gemini, VS Code, Git, VS Code Extension(Live Server)
> 이 문서는 "무엇을 만들지"가 아니라 "어떻게 진행할지"를 다룹니다. 요구사항 목록은 명세서 문서를 참고하세요.

>  Gemini Canvas 안내
>
>  Gemini Canvas는 사용량이 무한하지 않습니다. 사용 중 할당량이 다 되면:
>
> 1. Canvas가 아닌 **일반 채팅**으로 전환해서 코드 생성을 요청합니다.
> 2. VS Code에서 새 html 파일을 생성한 뒤, 생성된 코드를 붙여넣습니다.
> 3. 이 방식으로 이어서 개발하면 지속적인 작업이 가능합니다.

---

## 1. 실습 진행 방식

### Gemini Canvas 사용량이 충분한 경우
1. Gemini Canvas에 필요한 부분의 코드 생성을 요청합니다.
2. 요청한 수정이 잘 적용됐는지 확인하고 파일을 다운로드합니다.
3. 파일을 복사해 기존 Git 관리 파일에 덮어씌웁니다. (파일 교체)
4. Git으로 커밋을 남깁니다.
5. 1번부터 반복합니다.

### Gemini Canvas 사용량이 끝난 경우
1. 일반 Gemini Chatting으로 전체 코드 생성을 요청합니다.
2. 생성된 코드 전체를 복사해 기존 파일에 붙여넣습니다.
3. 파일을 직접 실행해 잘 보이는지 확인합니다.
4. 이상이 없으면 Git 커밋 후 1번을 반복합니다.
5. 문제가 있으면 코드 수정을 다시 요청하고 2번부터 반복합니다.

---

## 2. Git 버전 관리 방법

> 💡 **왜 필요한가?** 기능을 추가한 뒤 문제가 생겼을 때, 커밋 기록을 보고 "기능 추가 전" 시점으로 쉽게 되돌릴 수 있습니다. 파일을 여러 개 복사해두는 것보다 훨씬 깔끔합니다.

**진행 방법**
1. 새 폴더를 만들고, Canvas에서 만든 1일차 결과물을 다운로드해 넣습니다.
2. `git init` 후 `add`, `commit` 순서로 명령어를 bash 터미널에 작성합니다.
3. 커밋 메시지는 "1일차 기본형 완성"처럼 간단히 적습니다.
4. 이후 기능을 추가할 때마다 최신 코드를 같은 폴더에 덮어쓰고 다시 `add`, `commit`합니다. (`git init`은 최초 1회만)
5. "다크모드 추가", "스크롤 진행률 바 추가"처럼 무엇을 바꿨는지 알아볼 수 있게 커밋 메시지를 남깁니다.

**버전 관리 예시**

```
v1: 1일차 기본형 완성 (최초 커밋)
v2: 다크모드 버튼 추가
v3: 스크롤 진행률 바 추가
v4: 파티클 애니메이션 추가
```



## 3. 요청 예시 (프롬프트)

### 반응형 디자인 점검

```
"지금 만든 페이지를 모바일 화면 크기에서 확인했을 때 어색하거나 깨지는 부분이 있는지 점검해줘."

"있다면 자연스럽게 보이도록 반응형으로 수정해줘."
```



### 디자인 수정
```
"페이지 상단에 다크모드/라이트모드를 전환할 수 있는 버튼을 추가해줘."

"버튼을 누르면 배경색과 글자색이 자연스럽게 바뀌게 해줘."
```



### 애니메이션 추가

```
"배경에 은은하게 움직이는 작은 점(파티클) 효과를 추가해줘."

"너무 화려하지 않고, 있는 듯 없는 듯 자연스러운 느낌으로 해줘."
```



### 상호작용 기능

```
"사용자가 간단하게 작성할 수 있는 댓글 작성 기능을 페이지 하단에 추가해줘."
```



### 게이미피케이션

```
"지금 만든 페이지에 스크롤할 때마다 진행률이 올라가는 게이지를 우측 상단에 추가해줘."

"페이지 내용과 관련된 짧은 스피드 퀴즈를 별도 섹션으로 넣어줘."

"모은 포인트에 따라 브론즈/실버/골드 등급이나 재미있는 칭호를 보여줘."
```



### 아이디어 도출

```
"지금까지 만든 페이지를 쭉 살펴보고, 더 추가하면 좋을 것 같은 기능이나 아쉬운 부분을 3가지 정도 제안해줘."

"각 제안이 어떤 점에서 좋아지는지 이유도 간단히 설명해줘."

"그중에서 내 페이지 컨셉과 가장 잘 어울리는 것 하나를 추천해줘."
```

## 4. 진행 팁

- 쉬운 것부터 차근차근 시도하고, 시간이 남으면 도전적인 것도 해보세요.
- 오늘은 "완성"보다 "시도"가 더 중요합니다.
- 모든 것을 한 번에 요청하면 수정이 부정확할 수 있으니, **작은 단위로** 나눠서 요청하세요.
- 여기 없는 아이디어라도 스스로 도출해서 도전해봐도 좋습니다.

## 5. 진행 체크리스트

> 전부 체크하지 못해도 괜찮습니다. 하나라도 새로 시도했다면 충분합니다.

- [ ] Git으로 1일차 결과물의 버전을 기록해두었는가
- [ ] 심화 시작 전, 모바일 화면에서 먼저 확인했는가
- [ ] "디자인 변경"에서 1개 이상 추가했는가
- [ ] "애니메이션"에서 1개 이상 추가했는가
- [ ] 도전 아이디어 혹은 추가로 적용해보고 싶은 아이디어가 있는가
- [ ] 추가한 요소가 원래 내용을 가리지 않는지 확인했는가
- [ ] 애니메이션이 너무 어지럽지 않고 부드러운지 확인했는가
- [ ] 심화 기능 추가 후, 모바일 화면에서도 다시 깨지지 않는지 확인했는가
- [ ] 기능을 추가할 때마다 Git으로 커밋을 남겼는가

## 6. 용어 설명

- **반응형 디자인**: PC, 태블릿, 스마트폰처럼 화면 크기가 달라져도 레이아웃이 자연스럽게 맞춰지는 디자인 방식. "모바일 대응"이라고도 부릅니다.
- **게이미피케이션 (Gamification)**: 게임이 아닌 서비스에 포인트, 진행률, 배지처럼 게임적인 요소를 넣어서 재미와 몰입감을 주는 기법.
- **CSS 애니메이션**: 버튼이 눌리거나 요소가 스르륵 나타나는 등, 코드로 만드는 화면 움직임 효과.
- **패럴랙스**: 스크롤할 때 배경과 앞쪽 내용이 서로 다른 속도로 움직여서 입체감을 주는 효과.
- **localStorage**: 브라우저에 데이터를 저장해두는 기능. 새로고침하거나 나중에 다시 접속해도 저장된 내용이 남아있게 해줍니다.
- **Git / 커밋 (Commit)**: Git은 코드가 바뀔 때마다 그 시점을 기록해두는 버전 관리 도구입니다. "커밋"은 지금 상태를 하나의 저장 지점으로 남기는 것을 말하며, 나중에 특정 커밋 시점으로 되돌아갈 수 있습니다.
- **버전 (Version)**: 코드가 특정 시점에 어떤 상태였는지를 나타내는 스냅샷. Git에서는 커밋 하나하나가 곧 하나의 버전이며, v1 → v2 → v3처럼 순서대로 번호를 붙여 구분하기도 합니다.
